## 1. 测试基线

### 1.1 当前产品规则

- 登录页不展示预置的个人姓名或演示账号。
- 用户注册后，账号持久化到 `server/data/zhixueban.json` 的 `accounts` 表，密码仅保存 `scrypt` 哈希。
- 登录后的角色由账号决定。教师和学生不能通过顶部开关切换身份，也不能通过地址栏进入另一角色的工作台。
- 注册学生自动加入演示班级，可打开个人学习页。注册教师可使用演示班级完成教学流程。
- 主测验链路是：教师按四种题型出卷，发布，学生作答，自动评分，生成真实作答报告和错因。
- AI 不可用时，分层练习和在线测验必须使用本地题库或模板继续完成主流程。

### 1.2 文档同步项

`docs/ACCEPTANCE.md` 和 `docs/REQUIREMENTS.md` 中仍有“教师⇄学生角色切换”的旧描述。本文件以当前实现为准，相关旧验收项应更新为“账号角色固定，越权路由重定向”。

### 1.4 优先级和结果

| 标记 | 含义 | 发布判断 |
|---|---|---|
| P0 | 主路径、数据安全、登录或核心接口 | 必须通过 |
| P1 | 关键体验、异常处理、可用性 | 允许少量已知问题并有明确修复计划 |
| P2 | 体验增强、兼容性扩展 | 不阻断演示 |
| 结果 | Pass / Fail / Blocked / Not Run | 每次执行必须填入实际结果、证据和缺陷号 |

### 1.5 执行约束

1. 每个会修改数据的套件开始前执行 seed，避免前序数据影响结果。
2. 用唯一账号测试注册，例如 `qa-student-时间戳`；套件结束后再次 seed 清理测试账号。
3. AI 用例必须分别覆盖“可用”与“不可用或超时”两种状态，不能只验证 `/api/ai/status`。
4. 涉及学生数据的接口必须用 A 学生和 B 学生交叉调用，验证不会读取或写入对方数据。
5. UI 用例应同时记录截图、网络请求和页面 URL。接口用例应记录请求体、状态码和关键响应字段。
